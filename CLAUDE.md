## What is Speckle Automate?

Speckle Automate is a CI/CD platform that runs your custom code automatically whenever a new version of a Speckle model is published. It is an **Enterprise-only** offering on `app.speckle.systems`.

- **Function** = the code you write (Python or C#), stored in a GitHub repo.
- **Automation** = the configuration that links a Function to a model trigger ("when model X gets a new version, run Function Y with these parameters").
- **No server required** — Automate runs on Speckle's infrastructure in Linux Docker containers (CPU-only, no GPU).

### How deployment works

1. You push code to a GitHub repo.
2. You create a GitHub Release (e.g., tag `v1.0.0`).
3. A **GitHub Action** (`.github/workflows/main.yml`, auto-generated by Speckle's wizard) builds your Docker image and sends it to Speckle.
4. When a model version is published, **Speckle's servers** run your container.
5. Results appear in the Speckle web UI.

GitHub Actions = **build & deploy** pipeline. Speckle = **execution** runtime.

## Entry Point Pattern

Every Python Automate function follows this pattern in `main.py`:

```python
from speckle_automate import (
    AutomateBase,
    AutomationContext,
    execute_automate_function,
)
from pydantic import Field

# 1. Define inputs (generates UI form in Speckle)
class FunctionInputs(AutomateBase):
    # ... your input fields ...
    pass

# 2. Define your function
def automate_function(
    automate_context: AutomationContext,
    function_inputs: FunctionInputs,
) -> None:
    # ... your business logic ...
    pass

# 3. Wire it up
if __name__ == "__main__":
    execute_automate_function(automate_function, FunctionInputs)
```

## Function Inputs

Inputs subclass `AutomateBase` (which extends Pydantic's `BaseModel`). They auto-generate a web form in the Speckle UI.

### Supported types

| Type        | Python       | Use Case                     |
|-------------|--------------|------------------------------|
| Text        | `str`        | Names, descriptions, IDs     |
| Number      | `float`      | Measurements, thresholds     |
| Integer     | `int`        | Counts, indices              |
| Boolean     | `bool`       | Toggle options               |
| Selection   | `Enum`       | Dropdown choices             |
| Secret      | `SecretStr`  | API keys, passwords          |

### Validation attributes

| Validation | Python                           |
|------------|----------------------------------|
| Required   | `Field(...)`                     |
| Length     | `min_length=1, max_length=100`   |
| Range      | `gt=0, lt=100`                   |
| Pattern    | `pattern=r"^[A-Z]+$"`           |

### Example: All input types

```python
from speckle_automate import AutomateBase
from pydantic import Field, SecretStr
from enum import Enum

class AnalysisType(str, Enum):
    STRUCTURAL = "structural"
    THERMAL = "thermal"
    COST = "cost"

class FunctionInputs(AutomateBase):
    # Text input
    project_name: str = Field(
        title="Project Name",
        description="Enter the project identifier"
    )

    # Numeric input with default and validation
    max_length: float = Field(
        default=10.0,
        title="Maximum Length (m)",
        description="Maximum allowed length in meters",
        gt=0
    )

    # Boolean toggle
    include_analysis: bool = Field(
        default=True,
        title="Include Analysis",
        description="Whether to perform detailed analysis"
    )

    # Dropdown selection (from Enum)
    analysis_type: AnalysisType = Field(
        default=AnalysisType.STRUCTURAL,
        title="Analysis Type",
        description="Select the type of analysis to perform"
    )

    # Secret (encrypted, obfuscated in UI)
    api_key: SecretStr = Field(
        title="External API Key",
        description="Your external service API key"
    )
```

## AutomationContext — Your Interface to Speckle

The `automate_context` parameter gives you everything you need at runtime.

### Properties

| Property                | Type                | Description                                                |
|-------------------------|---------------------|------------------------------------------------------------|
| `automation_run_data`   | `AutomationRunData` | Metadata: project_id, automation_id, run IDs, triggers     |
| `speckle_client`        | `SpeckleClient`     | Pre-authenticated client for full SpecklePy SDK access     |
| `server_transport`      | Transport           | Usually not needed directly — SDK abstracts it             |

### AutomationRunData properties

- `project_id` — the project ID
- `speckle_server_url` — e.g., `https://app.speckle.systems/`
- `automation_id` — the automation this function belongs to
- `automation_run_id` — ID of this specific run
- `function_run_id` — ID of this function execution
- `triggers` — list of triggers (currently single model ID)

### Methods

#### Getting model data

```python
version_root_object = automate_context.receive_version()
# Returns a Base object — the root of the model's object tree
```

#### Reporting run status (MUST call one of these)

```python
automate_context.mark_run_success("All checks passed")
automate_context.mark_run_failed("3 beams exceed stress limit")
automate_context.mark_run_exception("Unexpected error: division by zero")
```

- `mark_run_success` — function completed successfully
- `mark_run_failed` — business logic failure (e.g., validation failed). This is a legitimate result.
- `mark_run_exception` — code error or unexpected crash

**Only the last status call is recorded.** Don't call multiple status methods.

#### Annotating objects (attach results to specific model elements)

```python
# Flag objects with errors
automate_context.attach_error_to_objects(
    category="Stress Check",
    affected_objects=failing_objects,  # Base or List[Base]
    message="Stress exceeds 250 MPa",
    metadata={"max_stress": 300.5}  # optional
)

# Add warnings
automate_context.attach_warning_to_objects(
    category="Performance",
    affected_objects=warn_objects,
    message="Near stress limit"
)

# Mark as successful
automate_context.attach_success_to_objects(
    category="Compliance",
    affected_objects=passing_objects,
    message="Within acceptable range"
)

# Informational notes
automate_context.attach_info_to_objects(
    category="Measurement",
    affected_objects=measured_objects,
    message="Measured length: 12.5m"
)
```

**Method signature** (same for all `attach_*_to_objects`):
```python
attach_error_to_objects(
    category: str,
    affected_objects: Union[Base, List[Base]],  # NOT string IDs
    message: Optional[str] = None,
    metadata: Optional[Dict[str, Any]] = None,
    visual_overrides: Optional[Dict[str, Any]] = None,
) -> None
```

**Best practices for annotations:**
- Use categories to group related annotations
- DON'T create one annotation per object — group related objects together
- Categories can represent severity, type, or description
- `metadata` is a key-value dict for arbitrary extra data
- `affected_objects` accepts Base objects directly, NOT string IDs

**Experimental gradient visualization:**
```python
automate_context.attach_info_to_objects(
    category="Gradient Visualization",
    metadata={"gradient": True, "gradientValues": gradient_values},
    message="Values applied to objects in list order",
    affected_objects=all_objects,
)
```

#### Creating new model versions

```python
automate_context.create_new_version_in_project(
    root_object=root,          # Base object to publish
    model_id="target-model-id",  # target model ID (from URL)
    version_message="Analysis results added",
)
# Returns: specklepy.core.api.models.current.Version
```

#### Saving file artefacts (reports, exports)

```python
automate_context.store_file_result("/tmp/report.pdf")
# Returns: str (the file URL)
```

#### Context views

```python
automate_context.context_view = "..."  # URL schema for custom view state in UI
```

## Working with Speckle Data

Speckle models are **directed acyclic graphs (DAGs)**:
- Objects reference other objects (parent → child)
- Properties can be primitive values or object references
- Common properties: `elements`, `parameters`, `units`, `applicationId`, `speckle_type`

### Flattening the object tree

The template includes a `flatten.py` utility. Use `flatten_base()` to iterate all objects:

```python
from flatten import flatten_base

version_root_object = automate_context.receive_version()

# Get all objects of a specific type
walls = [
    obj for obj in flatten_base(version_root_object)
    if obj.speckle_type == "Objects.BuiltElements.Wall"
]

# Get all beams
beams = [
    obj for obj in flatten_base(version_root_object)
    if obj.speckle_type == "Objects.BuiltElements.Beam"
]
```

### Common speckle_type values (from Revit/BIM connectors)

- `Objects.BuiltElements.Wall`
- `Objects.BuiltElements.Floor`
- `Objects.BuiltElements.Beam`
- `Objects.BuiltElements.Column`
- `Objects.BuiltElements.Room`
- `Objects.BuiltElements.Roof`
- `Objects.BuiltElements.Door`
- `Objects.BuiltElements.Window`
- `Objects.BuiltElements.Duct`
- `Objects.BuiltElements.Pipe`
- `Objects.Geometry.Mesh`
- `Objects.Geometry.Line`
- `Objects.Geometry.Point`

### Accessing object properties

```python
for wall in walls:
    # Direct properties
    wall_id = wall.id
    wall_type = wall.speckle_type

    # Parameters (BIM data)
    if hasattr(wall, "parameters"):
        params = wall.parameters
        # Access specific parameter values

    # Geometry
    if hasattr(wall, "displayValue"):
        meshes = wall.displayValue  # list of Mesh objects

    # Application ID (original ID from source app)
    if hasattr(wall, "applicationId"):
        original_id = wall.applicationId
```

## Complete Function Example

```python
from speckle_automate import (
    AutomateBase,
    AutomationContext,
    execute_automate_function,
)
from pydantic import Field
from flatten import flatten_base


class FunctionInputs(AutomateBase):
    forbidden_speckle_type: str = Field(
        title="Forbidden speckle type",
        description="Objects with this speckle_type will be flagged as errors.",
    )


def automate_function(
    automate_context: AutomationContext,
    function_inputs: FunctionInputs,
) -> None:
    """Flag objects of a forbidden type."""

    # 1. Receive the model data
    version_root_object = automate_context.receive_version()

    # 2. Find forbidden objects
    forbidden_objects = [
        obj
        for obj in flatten_base(version_root_object)
        if obj.speckle_type == function_inputs.forbidden_speckle_type
    ]

    count = len(forbidden_objects)

    if count > 0:
        # 3a. Attach errors to specific objects
        automate_context.attach_error_to_objects(
            category="Forbidden Type",
            affected_objects=forbidden_objects,
            message=f"Found {count} objects of forbidden type: "
                    f"{function_inputs.forbidden_speckle_type}",
        )
        automate_context.mark_run_failed(
            f"Found {count} objects of forbidden type "
            f"'{function_inputs.forbidden_speckle_type}'"
        )
    else:
        # 3b. Report success
        automate_context.mark_run_success(
            f"No objects of type '{function_inputs.forbidden_speckle_type}' found."
        )


if __name__ == "__main__":
    execute_automate_function(automate_function, FunctionInputs)
```

## Adding Dependencies

Edit `pyproject.toml`:

```toml
[project]
dependencies = [
    "specklepy==3.0.0",
    "pandas==2.1.0",       # Add runtime deps here
    "numpy>=1.24.0",
]

[project.optional-dependencies]
dev = [
    "pytest",
    "black",
    "pytest-mock",
]
```

Rule of thumb: if `main.py` imports it → `dependencies`. If it's a dev tool → `dev`.

### Test automations

Test automations are sandboxed environments that:
- Are NOT triggered by project data changes
- Cannot be manually triggered via UI
- Allow you to run your function locally while connecting to real Speckle data
- Results are visible on the automation page

## Releasing / Deploying

1. Make your changes, push to GitHub.
2. Create a new **GitHub Release** (use semver: `v1.0.0`, `v1.1.0`, etc.).
3. The GitHub Action automatically:
   - Builds a Docker image
   - Extracts and validates input schema
   - Sends compiled function to Speckle Automate
4. Your new version appears in the Functions Library.

Monitor the build in your repo's **Actions** tab.

## Infrastructure Constraints

- **CPU only** — no GPU compute (yet)
- **Linux containers** — no Windows runtime (yet)
- **GitHub is currently the only supported VCS** (alternatives planned)
- **No sandbox restrictions** — your function can make external API calls, use any dependencies
- **Speckle monitors containers** for misuse (crypto mining, spam, etc.) — zero tolerance
- **No inbound calls** to Speckle infrastructure — all automation must be configured via the Speckle UI
- **Token-based auth only** for third-party services (no OAuth2 flows yet)
- **Functions cannot be deleted** once created (contact Speckle team)
- **Updating function version in automation loses previously configured input values**
